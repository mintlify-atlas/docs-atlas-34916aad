title: "Phasic Policy Gradient (PPG) for ProcGen Environments"
description: "Improves PPO sample efficiency on ProcGen by separating policy and value training into distinct phases with an auxiliary distillation objective."
---

Phasic Policy Gradient (PPG) improves on PPO by separating policy optimization and value function training into two distinct phases. During the **policy phase**, PPO updates run as normal and all experience is saved to a replay buffer. During the **auxiliary phase**, the replay buffer is used to further train the value function and distill knowledge from the current policy into an auxiliary head, controlled by a behavioral cloning coefficient.

This separation prevents conflicting gradients from the policy and value objectives and improves sample efficiency on the ProcGen benchmark, at the cost of slower wall-clock time compared to standard PPO.

**Original paper:**

- [Phasic Policy Gradient](https://arxiv.org/abs/2009.04416) — Cobbe et al., 2021

**Reference implementation:**

- [openai/phasic-policy-gradient](https://github.com/openai/phasic-policy-gradient)

This implementation targets the ProcGen benchmark, which uses procedurally generated 64×64 RGB image observations and discrete action spaces.

### Key implementation details

`ppg_procgen.py` includes several implementation details from the original code that are not described in the paper:

1. **Full rollout sampling in the auxiliary phase** — Instead of randomly sampling individual observations, the auxiliary phase samples full rollouts of 256 steps from the replay buffer. This gives a meaningful performance boost.

2. **Batch-level advantage normalization** — Advantages are normalized over the entire batch before PPO updates, rather than per minibatch.

3. **Normalized network initialization** — All layers use a specific normalized initialization scheme:
   - Weights are initialized with Kaiming Uniform, then divided by their L2 norm.
   - Scale factors differ by layer: value/policy/auxiliary heads use `0.1`, the fully connected layer after the last conv uses `1.4`, and conv layers use approximately `0.638`.

4. **Adam ε parameter** — Set to `1e-8` (PyTorch default) rather than `1e-5` used in PPO.

5. **Consistent γ in reward normalization** — `gamma=0.999` is used in the `NormalizeReward` wrapper to match the PPO gamma, avoiding a mismatch present in earlier implementations.

<Note>
  The original PPG paper reports results on the `hard` distribution. CleanRL benchmarks run the `easy` distribution to reduce compute cost. The original code (`openai/phasic-policy-gradient`) is also run on `easy` mode for comparison.
</Note>

### Differences from the original implementation

- CleanRL does not support LSTM (the original does).
- CleanRL uses a single optimizer for policy and auxiliary phases (the original uses separate optimizers; the difference is small in practice).
- CleanRL does not support multi-GPU training.

### Experiment results

| Environment | `ppg_procgen.py` | `ppo_procgen.py` | openai/phasic-policy-gradient (easy) |
|-------------|------------------|------------------|---------------------------------------|
| Starpilot (easy) | 34.82 ± 13.77 | 32.47 ± 11.21 | 42.01 ± 9.59 |
| Bossfight (easy) | 10.78 ± 1.90 | 9.63 ± 2.35 | 10.71 ± 2.05 |
| Bigfish (easy) | 24.23 ± 10.73 | 16.80 ± 9.49 | 15.94 ± 10.80 |

Results are averaged over 25M timesteps.
